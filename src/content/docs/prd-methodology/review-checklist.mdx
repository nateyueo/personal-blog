## 评审前自查清单

在发起评审会之前，对照以下清单逐项确认：

### 📋 文档完整性

- [ ] 文档信息（版本/作者/日期/状态）是否完整？
- [ ] 执行摘要是否能在 30 秒内让读者理解要做什么？
- [ ] 背景与目标是否有数据支撑？
- [ ] 用户画像是否有真实调研依据？

### 🎯 需求清晰度

- [ ] 每个功能需求是否有明确的验收标准？
- [ ] 优先级是否合理分层？（P0/P1/P2）
- [ ] 是否明确列出了「不做什么」（scope out）？
- [ ] 边界条件和异常流程是否考虑到位？

### 🔗 依赖与约束

- [ ] 技术依赖是否已与开发团队确认可行？
- [ ] 第三方依赖（API/服务/数据）是否明确？
- [ ] 时间约束是否合理？
- [ ] 是否有法律/合规/安全约束需要注意？

### 📐 设计交付

- [ ] 是否提供了原型或线框图？
- [ ] 核心用户流程是否有流程图？
- [ ] 文案/副本是否已确认？
- [ ] 空状态/错误状态/加载状态是否已覆盖？

### 📊 度量与验证

- [ ] 成功指标（KPI）是否可量化？
- [ ] 数据埋点需求是否已列出？
- [ ] A/B 测试方案是否已规划？（如适用）

## 评审会要点

评审会上重点讨论：

1. **需求背景** — 确保所有人理解「为什么做」
2. **核心流程** — Walk through 主要用户路径
3. **技术风险** — 开发团队反馈实现难点
4. **排期与资源** — 确认交付时间线

> 💡 评审不是走过场。如果评审中发现重大问题，应该暂停并安排二次评审，而非带着问题进入开发。
