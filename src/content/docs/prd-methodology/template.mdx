## PRD 标准模板

以下是一份经过实战验证的 PRD 模板结构，可根据项目规模增减章节。

### 1. 文档信息

```markdown
| 字段 | 内容 |
|------|------|
| 文档名称 | [产品/功能名称] PRD |
| 版本 | v1.0 |
| 作者 | [姓名] |
| 日期 | [YYYY-MM-DD] |
| 状态 | 草稿 / 评审中 / 已发布 |
```

### 2. 执行摘要（Executive Summary）

用 2-3 句话概括：

- **做什么**：产品/功能的核心定义
- **为什么**：业务目标与用户价值
- **怎么验证**：关键成功指标

### 3. 背景与目标

**编写清单：**

- [ ] 项目背景是否清晰？
- [ ] 业务目标是否可量化？
- [ ] 用户痛点是否有数据支撑？
- [ ] 是否明确了非目标（scope out）？

### 4. 用户画像与场景

描述核心用户群体：

| 用户角色 | 核心需求 | 关键场景 | 使用频率 |
|----------|----------|----------|----------|
| [角色A] | [需求] | [场景描述] | [高/中/低] |

### 5. 功能需求（按优先级）

使用 MoSCoW 方法分级：

- **Must Have（P0）**：不做就不能上线
- **Should Have（P1）**：强烈期望，但有变通方案
- **Could Have（P2）**：锦上添花
- **Won't Have（本期不做）**：明确排除

### 6. 非功能需求

- 性能要求（响应时间、并发量）
- 安全要求（认证、授权、数据保护）
- 兼容性要求（浏览器、设备、系统版本）
- 可用性要求（SLA 目标）

### 7. 交互设计

- 原型链接或线框图
- 核心用户流程图
- 边界条件与异常处理

## 常见问题

### Q: PRD 要写多详细？

根据团队成熟度调整。核心原则是：**让读者能无歧义地理解你的意图**。

### Q: PRD 要多久更新一次？

建议跟随迭代节奏。每个 Sprint 开始前 review，结束后更新已交付部分的状态。
