## 完整编写流程

PRD 编写并非一蹴而就，建议遵循以下流程：

### Phase 1: 需求收集

- 用户调研（访谈、问卷、行为数据）
- 竞品分析
- 业务目标对齐

### Phase 2: 需求整理

- 需求分类与优先级排序
- 用户故事编写
- 技术可行性初评

### Phase 3: 文档编写

- 按模板填充内容
- 添加原型/线框图引用
- 补充边界条件与异常流程

### Phase 4: 评审与迭代

- 内部评审（产品团队）
- 跨团队评审（设计 + 开发）
- 根据反馈迭代优化

### Phase 5: 交付与跟踪

- 正式交付开发团队
- 建立变更管理机制
- 持续跟踪实现进度

## 流程要点

> 🔑 核心原则：PRD 是**活文档**，需要随项目推进持续更新，而非一次性交付。

## 常见误区

1. ❌ 把 PRD 当作「规格说明书」— 应该关注「为什么」而非仅描述「是什么」
2. ❌ 追求面面俱到 — 应该聚焦核心路径，逐步细化
3. ❌ 忽视非功能需求 — 性能、安全、可维护性同等重要
